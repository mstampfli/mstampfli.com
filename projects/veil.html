veil

A per-application network isolation engine. Veil confines any single process to its own Linux network namespace and routes its traffic through a composable chain of WireGuard, OpenVPN, Tor, SOCKS5 and HTTP hops, behind a kill switch that is verified, not assumed. One application, one exit path, with the rest of the host untouched. Written in Go, with a command-line interface and a desktop application.

The problem

A conventional VPN routes the entire host: every application shares one exit, and a single misbehaving process can deanonymize everything else on the link. Veil inverts that model. You scope the tunnel to one application and define exactly how its traffic leaves the machine, independent of whatever the rest of the host is doing.

Isolation

  • Per-application namespace, each profile launches its target process inside a dedicated Linux network namespace joined to the host by a veth pair.
  • No bare-link path, the process sees only the tunnel, never the host's physical interface or real IP address.
  • DNS confinement, name resolution is forced through the tunnel, eliminating the plaintext resolver leak that defeats most setups.
  • Unprivileged by design, a small veil-bridge helper holds cap_net_admin to create the veth pair and the firewall rules; the engine and your app never run as root. A rootless user-namespace mode is available too.

Composable routing

  • Backends, WireGuard (kernel or userspace), OpenVPN, the system Tor daemon, and SOCKS5 / HTTP proxies, all behind one backend interface.
  • Multi-hop, chain hops in series in any order, for example host → WireGuard → Tor → app, so no single provider sees both ends of the path.
  • Per-profile, each profile carries its own chain, so several applications can leave the machine through entirely separate paths at the same time.

Verified kill switch

Inside the namespace the firewall is default-deny: only the tunnel device and loopback may send traffic out, so a packet has nowhere to go the moment the tunnel is down. Veil does not just trust that rule, it runs an active reachability probe before handing the namespace to your application, confirming the only egress is the tunnel, and re-checks when the tunnel's state changes. If the tunnel drops, the app loses connectivity rather than leaking onto the open link.

Under the hood

  • Three binaries, veil (CLI), veil-gui (a GTK desktop app), and veil-bridge, the small cap_net_admin helper that does the privileged veth and firewall work.
  • Written in Go, with reproducible builds: the release compiles twice and byte-compares to prove the binary matches the source.
  • Signed self-updater, releases are Ed25519-signed and verified before an atomic in-place replace; an unsigned or tampered download is refused.
  • DNS, forced through the tunnel with an optional DoH proxy, so resolution is both confined and encrypted.

Built for

Privacy-critical workflows and authorized security research: red-team operations, threat-intelligence collection, malware-analysis sandboxing, and compartmentalized browsing. Per-application isolation means one identity can never bleed into another. For lawful, authorized use only.

Editions

Veil is open-core. The complete isolation engine is free and open source; the anti-detect and operational-security stack is a commercial, drop-in upgrade, the same binary unlocked by a signed offline license, with no migration and no separate install.

Veil Core

freeopen source

The complete network-isolation engine. No account, no profile limits, no telemetry.

  • Per-application Linux network-namespace isolation
  • Composable chains: WireGuard, OpenVPN, Tor, SOCKS5, HTTP
  • Multi-hop routing, unlimited profiles
  • Verified, fail-closed kill switch
  • DNS leak prevention inside the namespace
  • Command-line interface and GTK desktop application

Veil Pro

commercialanti-detect

Everything in Core, plus the anti-detect and operational-security stack for high-assurance research.

  • Per-profile browser personas, with a forge for unique identities
  • TCP, TLS and HTTP/2 fingerprint shaping (uTLS ClientHello, packet-level TCP options, HTTP/2 SETTINGS) aligned to the persona
  • Locked exit endpoints, pin a profile to an IP, city or ASN
  • Launch-window scheduling, behavioral jitter, CPU throttling
  • TLS-inspecting proxy with a managed certificate authority
  • Tor circuit control, including exit country and bridges
  • Signed offline licensing and a private, authenticated update channel

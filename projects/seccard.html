seccard

A security report card for any domain. Point it at a site and get a graded, actionable audit of TLS, HTTP security headers, and email/DNS authentication, with a concrete fix for every finding. Stdlib only, no API keys.

what it grades

  • TLS, chain and hostname validity, expiry, SANs, cipher strength, TLS 1.3 support, legacy TLS 1.0/1.1 acceptance, certificate key size and signature algorithm, and OCSP stapling.
  • HTTP security headers, HSTS, CSP (with a per-directive quality lint), X-Content-Type-Options, Referrer-Policy, Permissions-Policy, COOP/CORP, clickjacking, cookie flags, version disclosure, and the HTTP to HTTPS redirect.
  • Web hygiene, security.txt (RFC 9116) and whether the TRACE method is enabled.
  • Email and DNS auth, SPF (with a DNS-lookup budget), DMARC policy and reporting, DKIM, MX, CAA, DNSSEC, MTA-STS, and TLS-RPT, with mail checks skipped when there's no MX.

About 28 checks in all. Each is ok, warn, or FAIL, contributes to a 0 to 100 score that rolls up to a letter grade, with the exact fix printed for anything not green.

live demo

Pick a pre-scanned domain to see its real seccard report. (No live arbitrary-domain scanning here: a public "scan any host" endpoint would be an SSRF and abuse backend. seccard is a CLI you run against systems you're allowed to test; these are saved runs.)

Pick a domain.

example run

python3 seccard.py acme.example
$ python3 seccard.py acme.example

security report card: acme.example
overall: C  (71/100)

TLS
  [ok ]  TLS chain & hostname: valid certificate, negotiated TLSv1.3
  [ok ]  Cert expiry: 64 days remaining
  [FAIL] Legacy TLS: server still accepts TLS 1.0, TLS 1.1
         fix: disable TLS 1.0/1.1; require TLS 1.2+
HTTP headers
  [FAIL] Strict-Transport-Security: missing
         fix: add HSTS: max-age=63072000; includeSubDomains
  [warn] Content-Security-Policy: missing
  [ok ]  X-Content-Type-Options: nosniff
DNS & email auth
  [ok ]  SPF: v=spf1 include:_spf.google.com -all
  [FAIL] DMARC: no DMARC record
         fix: publish _dmarc TXT v=DMARC1; p=reject; rua=...
  [warn] CAA: no CAA record

Real story: I ran this against my own site, it found missing headers (and a bug in itself, see below), I fixed both, and the grade went from D to A.

two details I care about

  • It queries a public resolver, not the local one. Running it on my laptop first gave false negatives, because the laptop resolves through a split-horizon VPN resolver that hides the public records. A security audit tool must never trust that, so seccard asks 1.1.1.1.
  • There is no "scan any domain" web backend, on purpose. A public endpoint that fetches arbitrary hosts on demand is a server-side request forgery and abuse magnet. seccard is a CLI you run yourself, against systems you are allowed to test.

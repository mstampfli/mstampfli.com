codecard

A security report card for a whole codebase or deployment. Point it at a repo, a React app, or a Docker/Kubernetes setup and get one graded, actionable audit across source, config/IaC, secrets, and dependencies, with the exact fix for every finding. Deterministic first; the AI pass is optional, never required.

what it scans

  • Source, driven by semgrep's registry ruleset when installed, with a built-in per-language regex set (injection, insecure deserialization, weak crypto, TLS-off, ...) as the fully-offline fallback.
  • Config / IaC, Dockerfiles, Kubernetes, Terraform, and compose scanned by trivy for misconfigurations (running as root, :latest, privileged, ...), with hadolint adding Dockerfile depth.
  • Secrets, a built-in detector plus trivy, trufflehog, and gitleaks layered on for breadth and verified-secret confirmation, de-duplicated by file and line.
  • Dependencies, prioritized by real exploit intelligence, the differentiator (below).
  • Container images, --image TAG scans a built image's OS packages and bundled app dependencies, folded into the same ranked report.

It walks the whole tree, not a single file. External engines are use-if-present: codecard orchestrates the mature scanners when they're on PATH and falls back to its built-in rules when they aren't, then unifies and grades everything together.

the differentiator: which CVE first

Standard dependency scanning tells you "a CVE exists" and hands you a wall of them. codecard ranks every dependency CVE by CISA KEV (is it being exploited in the wild right now?) and FIRST EPSS (how likely is exploitation?), so the ones that actually matter float to the top. An actively-exploited dependency caps the grade. trivy and OSV find the vulnerabilities; codecard adds the layer that decides what to fix first, which is the part those tools don't do.

engines, downloaded for you

The external scanners are optional, but you don't have to hunt them down. codecard --setup pulls semgrep, trivy, trufflehog, gitleaks, and hadolint straight from their official releases into a local cache, no sudo, and codecard finds them there automatically on the next run.

usage
python3 codecard.py --setup                       # one-time: fetch the engines
python3 codecard.py ./myproject                   # scan a whole repo
python3 codecard.py ./myproject --image app:1.2   # ... and its built image
python3 codecard.py ./myproject --md report.md    # shareable report
python3 codecard.py ./myproject --ai              # optional LLM triage pass

the optional AI pass

Off by default, and the tool is fully useful without it. With --ai, an LLM gets the deterministic findings plus a system prompt as context and does two things the pattern scanners can't: it triages every finding as a true or false positive with a reason from the code, and it reports the structural bugs scanners miss (broken authorization, IDOR, auth bypass, business-logic flaws). It runs through a pluggable backend, the Claude CLI, a local Ollama model for proprietary code, or any OpenAI-compatible API. AI findings are labeled to verify, because they are leads, not verdicts.

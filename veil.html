why / one app, one tunnel

Isolation scoped to the app, not the host

A conventional VPN routes the whole machine: every app shares one exit, and one chatty process can deanonymize the rest. veil scopes the tunnel to a single application.

Per-app isolation

Each profile runs its process in its own Linux network namespace — it sees only the tunnel, never the host's real interface, IP, or resolver.

Composable multi-hop

Chain WireGuard, OpenVPN, Tor, SOCKS5 and HTTP in any order (e.g. host → WireGuard → Tor → app), so no single provider sees both ends.

Verified kill switch

veil proves there is no route out except the tunnel before the app starts, and fails closed if the tunnel drops. It verifies; it doesn't assume.

Zero host capabilities

The engine runs unprivileged via user namespaces + a userspace uplink (pasta) — CapEff=0, no cap_net_admin, no daemon.

No telemetry

The free edition makes zero network calls: no analytics, no update pings, no phone-home. Pro validates its license offline; its only network calls are manual updates and one activation log on install (a one-way machine hash, no usage data, no device cap). Nothing is tracked.

GUI + CLI + Docker

A GTK desktop app and a scriptable CLI, plus a hardened capless Docker image (with an optional headful-browser variant).

editions / free and pro

Two editions, one engine

veil is open-core. The complete isolation engine is free and source-available; the anti-detect and operational-security stack is a commercial upgrade — the same engine, unlocked by a signed offline license.

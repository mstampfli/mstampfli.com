-1 % 768 == 255

A WGSL compute shader on NVIDIA returns 255 where the spec says -1. It looks exactly like an NVIDIA driver bug. It isn't, and one clause in the Vulkan spec flips who is at fault. A short detective story about almost blaming the wrong layer.

[ tl;dr ]

WGSL guarantees -1 % 768 == -1, yet a WebGPU shader on NVIDIA returns 255. naga lowers signed % to SPIR-V OpSRem, and in the Vulkan environment OpSRem with a negative operand is poison (undefined) unless VK_KHR_maintenance8 is enabled, which the toolchain never requests. So NVIDIA is conformant; the bug is that the shader toolchain emits undefined-behaviour SPIR-V for a value WGSL fully defines. Fix: lower % as a - b * (a / b).

how i ran into it

My voxel engine stores the world in a wrap-around slot grid: the world coordinate is mapped into fixed storage with a positive modulo, pos_mod(world_voxel, N). Near spawn, perfect. Fly into negative world coordinates and the GPU drew the wrong chunk's terrain, geometry that wasn't really there and that the CPU raycast (correct Rust rem_euclid) never agreed existed. Every CPU unit test passed. The bug lived only on the GPU, only for negative inputs.

the fingerprint

-1 % 768 came back as 255, and (-1 as u32) % 768 == 255 exactly. Every wrong result matched (a as u32) % b: the negative dividend was being treated as unsigned. That number was not noise; it named the mechanism.

isolating the layer

A WGSL % flows WGSL --naga--> SPIR-V --driver--> machine code. naga translates to SPIR-V once, device-independently, so every driver gets the same bytes. I ran the identical shader (runtime operands, nothing const-folded) on every Vulkan device on the machine:

cross-driver differential (no maintenance8)
inputs              : [-1, -5, -768, -769, -1000, 0, 5, 768, 1000]
spec-correct  a%b   : [-1, -5,    0,   -1,  -232, 0, 5,   0,  232]

NVIDIA RTX 5060     : [255, 251, 256, 255, 24, ...]   WRONG
Intel Mesa (ANV)    : [-1, -5, 0, -1, -232, ...]      correct
llvmpipe (software) : [-1, -5, 0, -1, -232, ...]      correct

Same SPIR-V, only NVIDIA wrong. Dumping the bytes confirmed naga emits OpSRem, the signed remainder opcode, which is the right choice for WGSL %. So: not my code, not naga's opcode choice. Open and shut, an NVIDIA driver bug. Right?

the twist

Before reporting it, I checked whether the Vulkan spec actually requires the correct result. It doesn't, not by default. From the SPIR-V environment appendix:

[ vulkan spec ]

"If the maintenance8 feature is not enabled and if either operand to OpSRem and OpSMod instructions is negative the result is poison."

OpSRem on a negative operand is undefined in base Vulkan; the extension VK_KHR_maintenance8 is what makes it well-defined. So NVIDIA returning garbage without that extension is legal. The "obvious driver bug" wasn't one, until proven otherwise.

the decider

I wrote a raw-Vulkan reproducer that runs the shader on the same NVIDIA device twice, without and with VK_KHR_maintenance8 enabled:

maintenance8 A/B on the same device
NVIDIA a%b  (maintenance8 OFF): [255, 251, 256, ...]   WRONG   (legal poison)
NVIDIA a%b  (maintenance8 ON ): [-1, -5, 0, ...]       CORRECT

That settles it. Ask for defined semantics and NVIDIA is correct. NVIDIA is conformant. The bug is upstream: naga emits OpSRem without enabling maintenance8 and without guarding negatives, producing SPIR-V whose result is poison for the exact inputs WGSL says must yield a specific value. Mesa's drivers happen to define the behaviour; NVIDIA takes the latitude the spec grants.

the fix

OpSDiv is not in the poison clause (and NVIDIA computes / correctly without maintenance8), so signed % can be reconstructed poison-free:

wgsl
let r = a - b * (a / b);   // == a % b; correct on NVIDIA, no extension needed

naga already has a wrapper function for signed % and / (it guards divide-by-zero and INT_MIN % -1 to match WGSL). That wrapper is the natural place to emit the reconstruction instead of OpSRem, a small driver-gated change. It is now a pull request to wgpu; the negative-dividend test is merged into the WebGPU Conformance Test Suite, so the class of bug gets caught going forward.

why the twist matters

The first three steps "proved" it was NVIDIA. They were good evidence and completely misleading about fault. The lesson isn't "test on the GPU" (though do that). It's this: before you file a bug against an implementation, check whether the spec actually requires the behaviour you expected. SPIR-V core mandates the sign of OpSRem; the Vulkan environment relaxes it to poison, and the environment is what your app is held to. One sentence in an appendix moved this from "NVIDIA driver bug" to "toolchain emits undefined-behaviour SPIR-V", a different bug with a different owner and a different fix.

  • The fingerprint is data. (-1 as u32) % 768 == 255 named the mechanism immediately.
  • A cross-driver differential isolates the layer, not the fault. Two drivers agreeing and one disagreeing on identical bytes tells you where they differ, not who is wrong.
  • Undefined behaviour makes "wrong" legal. Check the spec's requirements, not just its definitions.

reproducer

Standalone, runs on any machine with Vulkan: a cross-driver differential, a SPIR-V opcode dump showing naga emits OpSRem, and the raw-Vulkan maintenance8 A/B that settles it.
